# 三星堆数字祭祀 AR 系统 — 视觉系统设计文档

> 文档定位:本系统「视觉追踪与渲染子系统」的设计规格。
> 适用范围:全栈 AR 主线(摄像头 → 人脸追踪 → 3D 青铜面具渲染 → 认主解锁 → 表情驱动)。
> 不在本文档范围:前端业务流程页、后端 API、桌宠、证书生成。本文档只管「视觉好不好看、追踪准不准、3D 特效够不够惊艳」。

---

## 1. 设计目标与「素」的根因诊断

### 1.1 目标
让三星堆青铜面具在浏览器 AR 中达到**影视级质感**,而非「抖音滤镜感」。评委现场戴上脸的那一秒,应当分辨不出「这是网页还是影视特效」。

### 1.2 为什么原方案看起来「素」
原计划路线为 `GLB 面具 + 2D 贴图 + emissive 改色`,存在三个根因:

| 根因 | 现象 | 病灶 |
|---|---|---|
| 材质错误 | 青铜灰扑扑、无金属光泽 | 用 MeshBasicMaterial 贴图,而非 PBR 金属材质;无环境反射 |
| 光照错误 | 面具扁平、无立体感 | 只有一个正面平行光,缺轮廓光与补光 |
| 动画错误 | 认主瞬间只是改数值 | 苏醒无时间线编排、无粒子、无 bloom 溢光 |

**核心判断**:「素」不在渲染精度,而在范式是查表贴图。本设计从材质、光照、后处理、动画四层重建视觉,并引入「认主解锁」范式让 468 关键点首次作为「解锁验证输入」而非「渲染锚点」。

## 2. 技术栈与可行性依据

| 组件 | 技术 | 依据 |
|---|---|---|
| 人脸追踪 | `@mediapipe/tasks-vision` FaceLandmarker | 官方支持 `outputFacialTransformationMatrixes`,直接输出 4×4 头部姿态矩阵,无需手写 solvePnP [$TRAE_REF](https://wenku.csdn.net/answer/5ni6pzwcttyo) |
| 3D 渲染 | three.js + EffectComposer + UnrealBloomPass | bloom 后处理是 three.js 标准插件,成熟可用 [$TRAE_REF](https://www.szxn.com/87289.html) |
| 动画编排 | gsap | 时间线编排,保证苏醒节奏感 |
| 青铜材质 | MeshStandardMaterial(PBR) + ShaderMaterial(铜锈层) | ShaderMaterial 是 three.js 标准做法 [$TRAE_REF](https://juejin.cn/post/7553569648131866667) |

**关键取舍**:浏览器端不手写 solvePnP。MediaPipe Tasks Vision 版直接输出 `facialTransformationMatrixes`(4×4 齐次矩阵),赋值给 Three.js Mesh 的 `matrix`。这是与原 Python 方案最大的技术差异,也是最稳的一条路——少一个自写算法环节,少一个翻车点。

**最小验证原型**:已在 `sanxingdui-ar-tracker/index.html` 编写可运行原型(程序化几何面具 + MediaPipe 矩阵贴合 + bloom + gsap 苏醒动画),用于在写实现计划前证实整条链路可行。

## 3. 追踪管线设计

### 3.1 数据流
```
摄像头(video) → MediaPipe FaceLandmarker.detectForVideo()
   ├─ facialTransformationMatrixes[0].data  (16 个浮点,行优先 4×4)
   ├─ faceBlendshapes (52 个表情系数)
   └─ faceLandmarks (468 个关键点,归一化坐标)
        │
        ▼
   坐标系转换(见 3.2)
        │
        ├──→ 渲染路:matrix → maskGroup.matrix (每帧,绕过 React,用 ref)
        └──→ 认主路:blendshapes/landmarks → 状态机判定 (低频,可入 state)
```

### 3.2 坐标系转换(关键且需现场校准)
MediaPipe 输出的矩阵在其自身归一化坐标系,转 Three.js 需处理:
- **镜像**:视频用 CSS `transform: scaleX(-1)` 镜像,矩阵的 x 分量需翻转(`elements[0/4/8/12] *= -1`)
- **缩放**:MediaPipe 矩阵的 scale 量级约为 1e-4,需放大到 Three.js 世界单位。放大系数需现场根据摄像头分辨率与 `camera.position.z` 调试,验证原型中先用 `abs(matrixData[14]) * 8` 估算,实际值待校准
- **平滑**:`position.lerp(target, 0.4)` + `quaternion.slerp(targetQuat, 0.4)`,0.4 为阻尼系数,消除逐帧抖动

> ⚠️ 待校准项:缩放系数与 z 深度映射需在真实摄像头下实测。若缩放抖动,改为基于 `faceLandmarks` 包围盒尺寸计算 scale,弃用矩阵 scale 分量。

### 3.3 性能策略(遵循 React 最佳实践)
- 468 关键点与矩阵属高频瞬态值,**用 `useRef` 不进 `useState`**(`rerender-use-ref-transient-values`)
- Three.js / MediaPipe / gsap 按 `bundle-dynamic-imports` 动态加载,首页不被 3D 库拖累
- 追踪循环用 `requestAnimationFrame`,检测用 `detectForVideo(video, performance.now())`,仅在 `video.currentTime` 变化时检测

## 4. 3D 渲染核心

### 4.1 材质:PBR 青铜
| 参数 | 值 | 理由 |
|---|---|---|
| `color` | `0x8a6d3b` | 青铜底色 |
| `metalness` | `0.9` | 青铜是金属,非石头 |
| `roughness` | `0.4` | 半光泽,有岁月感而非崭新 |
| `envMap` | HDR 环境贴图(Skybox) | 金属表面反射环境光,这是青铜「活起来」的关键——无 envMap 的金属呈死灰色 |
| `emissive` | 初始 `0x000000`,苏醒后渐变 | 见 §6 |

### 4.2 铜锈层(参数化生成的视觉落点)
在青铜基底上叠加第二层 `ShaderMaterial`:
- 用噪声纹理(程序化 simplex noise 或预生成铜锈 PNG)控制绿色斑块分布
- **决策向量驱动铜锈 alpha**:用户选择 → 铜锈覆盖率/分布参数 → shader uniform
- 这是「AI 生成感」的视觉落点:不是贴预制图,是参数驱动纹饰

### 4.3 光照:三点布光
| 光 | 类型 | 色温 | 位置 | 作用 |
|---|---|---|---|---|
| 主光 | DirectionalLight,强度 2.0 | 暖 `0xffd9a0` | (2,2,3) | 模拟祭祀火光,主照明 |
| 补光 | DirectionalLight,强度 0.8 | 冷 `0x6b8aff` | (-2,1,2) | 勾出另一侧轮廓 |
| 轮廓光 | DirectionalLight,强度 1.2 | 暖 `0xffaa55` | (0,1,-3) | 背后把面具从背景抠出 |
| 环境光 | AmbientLight,0.3 | `0x333344` | — | 暗部不死黑 |

> 三点布光是影视级 3D 的最低门槛,成本极低,效果是「评委一眼觉得专业」的分水岭。

### 4.4 后处理:Bloom 发光
- `EffectComposer` + `RenderPass` + `UnrealBloomPass`
- 参数:`strength=0.8, radius=0.4, threshold=0.3`
- `threshold=0.3` 保证只有 emissive 亮区(纵目、纹饰、粒子)溢光,青铜主体不糊
- `renderer.toneMapping = ACESFilmicToneMapping`,曝光 1.0

## 5. 面具模型来源策略

模型来源决定生成路线,是本系统唯一需提前拍板的风险点。三选一:

| 方案 | 形态生成能力 | 美术压力 | 风险 |
|---|---|---|---|
| 开源 GLB | 弱(无 morph) | 低 | 找不到带锚点的三星堆 GLB |
| 美工做带 morph 的 GLB | 强 | 高 | 48h 美工压力大 |
| **程序化几何(验证原型路线)** | 中(参数化形态有限) | 无 | 美术感依赖 shader |

**验证原型采用程序化几何**:扁球脸 + 圆柱纵目 + 装饰条,完全代码生成不依赖外部模型。若最终需更高美术感,替换为 GLB,材质与光照设计不变。

## 6. 认主解锁视觉状态机(创新钩子)

### 6.1 三态状态机
```
[扫描态] ──条件达标+稳定1.5s──> [验证态] ──通过──> [认主态]
   ↑                                                        │
   └──────────失败/超时回退──────────────────────────────────┘
```

| 态 | 视觉表现 | 持续 |
|---|---|---|
| 扫描态 | 面具半透明悬浮 + 扫描线 shader 动画,「审视」用户 | 等待条件 |
| 验证态 | 提示「睁眼/抬头」,检测 `eyeAspectRatio` + `pitch` 达标 | 1.5s 稳定 |
| 认主态 | 「附身」:悬浮→贴合,emissive 激活,粒子爆发 | 永驻 |

### 6.2 认主苏醒瞬间:三层动画叠加(gsap 编排)
- **第 1 层(0–0.5s)**:`emissive` 从黑渐变到神力色(纵目=暖金 `0xffaa33` / 云雷=青绿 / 金面=纯金)
- **第 2 层(0.3–0.8s)**:`metalness` 短暂降到 0.2 再回弹,面具表面闪过一道「非金属」光波,像神力注入
- **第 3 层(0.5–1.2s)**:粒子系统爆发(Three.js `Points` + 自定义 shader 粒子),青铜碎屑飞散 + 金光汇聚

### 6.3 认主条件检测(468 关键点首次作为验证输入)
- `eyeAspectRatio` = 关键点 159–145 垂直距离 / 33–133 水平距离,> 0.2 判定睁眼
- `pitch`:从 `facialTransformationMatrixes` 解算欧拉角(矩阵→四元数→欧拉),> 5° 判定抬头
- 条件达标后需**持续稳定 1.5s**,防误触

## 7. 表情驱动视觉联动

| 用户表情 | 检测信号 | 面具反馈 |
|---|---|---|
| 张嘴 | `mouthOpen` blendshape > 0.3 | 面具嘴部 emissive 亮起 / 装饰条发光 |
| 眨眼 | `eyeBlink` blendshape > 0.5 | 纵目 emissive 闪烁 |
| 微笑 | `mouthSmile` blendshape > 0.3 | 眼部亮度增强 |
| 头部转动 | matrix 的 yaw/pitch | 面具 3D 透视跟随(已由追踪保证) |

> 实现优先级:先做张嘴联动(最易检测、最直观),其余为时间允许的增量。

## 7.4 手势驱动神力降临(拓展:手部动作控制特效)

§7 表情联动是「面部输入驱动面具」,本节拓展为「手部输入驱动环境级特效」。认主完成后,用户用手势召唤「神力降临」——这是祭祀仪式的高潮落点,也是手部追踪首次作为「主动召唤输入」而非「被动渲染锚点」。

### 7.4.1 手势选型

| 候选手势 | 文化契合 | 检测稳定性 | 单摄像头可行性 | 结论 |
|---|---|---|---|---|
| 双手合十 | 高(祭祀礼拜) | 高 | 低(合十会遮挡脸部,人脸追踪丢失) | 否决 |
| 握拳 | 低 | 高 | 高 | 缺乏召唤感,否决 |
| **张开手掌(五指伸直)** | **高(展掌=召唤/献祭姿态)** | **高(5 指伸直几何特征强)** | **高(单手侧举不遮脸)** | **采用** |
| 食指中指竖起(剪刀) | 低(现代手势) | 高 | 高 | 与祭祀主题违和,否决 |

**采用张开手掌**:祭祀中「展掌」是召唤神力/献祭的标志性姿态,文化契合;5 指伸直的几何特征(指尖到腕距离 > PIP 关节到腕距离)稳定易检,误触率低。

### 7.4.2 数据流

```
摄像头(video) → MediaPipe HandLandmarker.detectForVideo()
   └─ landmarks[0] (21 个手部关键点,归一化坐标)
        │
        ▼
   isOpenPalm() 判定(5 指伸直)
        │
        ├──→ 张掌稳定 0.2s → divineTarget = 1(激活)
        └──→ 松手/手离镜头   → divineTarget = 0(淡出)
        │
        ▼
   updateDivineEffects():divineCurrent 平滑跟随 divineTarget
        │
        └──→ 神力降临四层特效强度 = divineCurrent
```

### 7.4.3 张掌检测算法

- 21 关键点:0 腕,1–4 拇指,5–8 食指,9–12 中指,13–16 无名指,17–20 小指
- **单指伸直判定**:`dist(指尖, 腕) > dist(PIP关节, 腕) × 1.1`(留 10% 余量防抖)
- **张掌判定**:四指(食/中/无名/小)均伸直 + 拇指横向伸展(指尖离腕 > MCP 离腕 × 1.1)
- **防抖**:张掌持续 0.2s 才激活神力,防瞬时误触

### 7.4.4 神力降临四层特效(divineCurrent 强度驱动)

| 层 | 特效 | 实现 | 强度映射 |
|---|---|---|---|
| 1 | 神圣光环 | 两个交叉旋转 `TorusGeometry`,`emissive` 神力色,加色 bloom 溢光 | `emissiveIntensity = 2.5 × v`,`opacity = v` |
| 2 | 神力光锥 | 面具后方 `ConeGeometry`(open-ended),加色混合半透明,营造神光辐射 | `opacity = 0.18 × v × 脉动` |
| 3 | 升腾金光粒子 | 240 个 `Points` + 自定义 shader,从面具下方持续上升、顶部循环重生 | `uIntensity = v`,`gl_PointSize ∝ v` |
| 4 | Bloom 增强 | `UnrealBloomPass.strength` 从 0.8 提升到 1.7 | `strength = 0.8 + 0.9 × v` |

> 神力色随 §4.2 决策向量切换:纵目=暖金 / 云雷=青绿 / 金面=纯金,与认主苏醒同源,保证视觉语言统一。

### 7.4.5 关键设计决策

- **HandLandmarker 用 CPU delegate**:FaceLandmarker 已占用 GPU(WebGL 上下文),若手部模型也用 GPU,两者同帧 `detectForVideo` 会竞争同一上下文导致互相踢掉(典型症状:手势一开,脸部追踪丢失)。手部模型仅 21 关键点(人脸 468 点),CPU 推理 10-15ms 可接受,彻底规避 GPU 冲突
- **手势检测始终运行,神力特效仅认主后触发**:HUD 实时显示手部/手势状态,让用户随时确认两者共存正常;神力降临作为认主成功的「奖励」(`ritualPhase === 'done'`),形成「先认主 → 再召唤」的仪式递进
- **独立时间戳**:手势检测用 `performance.now()` 重新取值,与人脸检测的时间戳序列分离,避免 MediaPipe 内部同时间戳调用的潜在干扰
- **独立 `divineGroup`**:跟随面具位置但不继承头部旋转,光环始终正面朝向镜头,避免随头部乱转破坏「神圣感」
- **隔帧检测**:降低双模型总开销(30fps 下手势 15Hz),hold 类手势对刷新率不敏感
- **指数平滑过渡**:`divineCurrent += (target - current) × dt × 6`,实现淡入淡出,避免特效硬切
- **兜底**:手部模型加载失败不阻塞主流程(§8.2 思想),HUD 显示「模型加载失败」,人脸追踪与表情联动照常工作

## 8. 性能预算与兜底

### 8.1 性能预算(单人脸,720p)
| 项 | 预算 |
|---|---|
| FaceLandmarker 检测 | ~8–12ms/帧 |
| HandLandmarker 检测(隔帧,§7.4) | ~5–8ms/隔帧 |
| Three.js 渲染 + bloom | ~5–8ms/帧 |
| 目标帧率 | ≥ 25 fps(30 fps 理想) |
| 像素比上限 | `min(devicePixelRatio, 2)`,防 4K 屏过载 |

### 8.2 兜底链
| 失败 | 兜底 |
|---|---|
| GLB 加载失败 | 降级为程序化几何面具(验证原型路线) |
| FaceLandmarker 失败 | 播放预录视频演示 |
| HandLandmarker 加载/检测失败 | 不阻塞主流程,HUD 提示「模型加载失败」,人脸追踪与表情联动照常(§7.4.5) |
| 认主条件检测失败 | 按钮「跳过认主」直接进佩戴态 |
| bloom 性能不足 | 关闭 bloom,仅保留 emissive;神力特效降级为无溢光的 emissive 光环 |

## 9. 验收标准

- 摄像头中转头,青铜面具跟随头部移动并保持 3D 透视(非平面贴纸)
- 青铜表面有金属光泽与环境反射,非死灰
- 纵目/emissive 区域有 bloom 溢光,暗背景中发光
- 认主瞬间有可见的三层动画(变色 + 光波 + 粒子)
- 张嘴时面具嘴部有可见反馈
- 认主后向镜头张开手掌,触发神力降临:光环旋转 + 升腾金光 + 光锥 + Bloom 增强(§7.4)
- 松手或手离镜头,神力特效平滑淡出
- 单人脸 ≥ 25 fps

## 10. 风险与未决项

| 项 | 状态 | 处置 |
|---|---|---|
| MediaPipe 矩阵缩放系数 | 待校准 | 真实摄像头下实测,必要时改用关键点包围盒算 scale |
| 程序化几何美术感 | 中 | 若不够惊艳,替换 GLB,材质/光照设计不变 |
| 粒子 shader 复杂度 | 待定 | 先用 Points + 简单纹理,复杂度超时则降级为 sprite |
| 认主条件误触率 | 待测 | 1.5s 稳定窗口 + 阈值现场调 |
| 双 GPU 模型同帧冲突(Face+Hand) | 待测 | 手势隔帧检测降低冲突窗口;若仍冲突,HandLandmarker 改 delegate:'CPU' |
| 张掌误触率 | 待测 | 0.2s 稳定窗口 + 1.1× 余量阈值现场调 |
