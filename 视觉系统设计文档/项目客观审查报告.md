## 1. 致命伤:手势系统是半成品 ★最严重

### 1.1 张掌 / 握拳 = 零特效,纯改文字

[index.html:1326-1333](file:///c:/Users/sunjunjie/Desktop/作业/111/sanxingdui-ar-tracker/index.html#L1326):

```javascript
if (g.gesture === 'palm') {
  action = '待机(张开)';      // ← 只赋一个字符串
  gestureHoldStart = 0;
} else if (g.gesture === 'fist') {
  action = '待机(握拳)';      // ← 只赋一个字符串
  gestureHoldStart = 0;
}
```

两个手势的全部副作用就是把 HUD 的「手势动作」行从"待机"改成"待机(张开)"/"待机(握拳)"。**没有触发任何视觉特效、没有改 emissive、没有粒子、没有 shader uniform 变化**。用户对着镜头张掌、握拳,屏幕上除了 HUD 一行小字变了,什么都看不到。

HUD 的 hint 还在承诺"张开手掌/握拳/捏合/指向 触发不同动作"([index.html:116](file:///c:/Users/sunjunjie/Desktop/作业/111/sanxingdui-ar-tracker/index.html#L116))——"捏合"和"指向"确实有动作(切色/爆发粒子),但"张掌"和"握拳"是空壳。这是**承诺了却没交付**的功能。

### 1.2 神力降临特效系统被删干净了,但 HUD 文案还在卖

- 全代码库搜索 `divineGroup` / `halo` / `ascendParticles` / `setDivineTarget` / `updateDivineEffects` / `divineCurrent` / `divineActive` —— **全部 0 匹配**。神力降临的光环、光锥、升腾粒子、强度驱动函数,整个系统不存在。
- 但 [index.html:657](file:///c:/Users/sunjunjie/Desktop/作业/111/sanxingdui-ar-tracker/index.html#L657) 认主完成提示文案仍是「向镜头张开手掌(五指伸直)召唤神力降临」,[index.html:790](file:///c:/Users/sunjunjie/Desktop/作业/111/sanxingdui-ar-tracker/index.html#L790) 注释仍写「张开手掌(五指伸直)触发神力降临」。

**结论**:神力降临系统在某个版本被整体删除,但触发它的入口(文案、手势映射预期)没清干净,留下一堆"指向不存在的函数"的注释和承诺。用户照文案张掌,什么降临都不会有。这是**功能与文案严重脱节的死代码残留**。

### 1.3 4 种手势实际只有 2 种有效

| 手势 | 代码行为 | 用户可见效果 |
|---|---|---|
| palm(张掌) | 赋字符串"待机(张开)" | **无** |
| fist(握拳) | 赋字符串"待机(握拳)" | **无** |
| pinch(捏合) | `cycleAffinityByGesture()` 切换 affinity + 改 emissive 颜色 | 有(颜色变) |
| point(指向) | `burstParticlesToward()` 定向粒子爆发 | 有(粒子飞) |

4 种手势里 2 种是空壳,占 50%。手势识别算法(`isOpenPalm`/`isFist`/`isPointing`/`isPinch`,[index.html:800-840](file:///c:/Users/sunjunjie/Desktop/作业/111/sanxingdui-ar-tracker/index.html#L800))写得挺全,识别了之后什么都不做,等于识别了个寂寞。

## 2. 代码组织:1412 行单文件,无模块化

### 2.1 全部逻辑塞在一个 `<script type="module">` 里

[index.html:139](file:///c:/Users/sunjunjie/Desktop/作业/111/sanxingdui-ar-tracker/index.html#L139) 到 [index.html:1450](file:///c:/Users/sunjunjie/Desktop/作业/111/sanxingdui-ar-tracker/index.html#L1450),约 1300 行 JS 全在 HTML 一个标签内。包含:Three.js 渲染初始化、MediaPipe 人脸/手部初始化、摄像头、4 种 shader 字符串、状态机、粒子系统、手势识别、2D 关键点绘制、HUD 更新、事件绑定。

后果:
- 无法对任何一块单独测试
- 改一行 shader 要在 1300 行里找模板字符串([index.html:267-308](file:///c:/Users/sunjunjie/Desktop/作业/111/sanxingdui-ar-tracker/index.html#L267) 嵌在 JS 里的 GLSL,无语法高亮)
- 全局变量遍地(`faceLandmarker`/`handLandmarker`/`three`/`ritualPhase`/`burstActive`/`prevGesture`...数十个),任何函数都能改任何状态,无封装

项目根目录没有任何 `src/` 目录,`package.json` 的 `scripts` 只有 `dev`/`build`/`preview`/`test`(test 是 `echo Error: no test specified`)。**零测试**。

### 2.2 全局状态散落,状态机靠字符串

```
ritualPhase = 'scan' | 'verify' | 'claim' | 'done'   // 字符串状态机
divineActive / burstActive / handFrameToggle / prevGesture / prevPosValid
lastDetected / lastHandLandmarks / scanEnterTimer / verifyStableStart

```

至少 15 个模块级 `let` 变量散落在文件各处,任何函数都能写它们。`ritualPhase` 用字符串比较(`=== 'done'`),无类型约束,打错一个字母(比如 `'dnoe'`)不会报错,直接静默失效。这种状态管理在 500 行还凑合,1400 行已经是定时炸弹。

## 3. 功能性硬伤

### 3.1 认主阈值过松,"抬头"动作形同虚设

[index.html:1259](file:///c:/Users/sunjunjie/Desktop/作业/111/sanxingdui-ar-tracker/index.html#L1259) `const ok = ear > 0.2 && pitch > 5`。`pitch > 5°` 太松——人正常平视摄像头,pitch 常在 3-8° 区间(取决于摄像头高度)。用户根本不需要刻意抬头,只要睁眼平视 1.5s 就过认主。所谓"仪式动作"在阈值层面就被消解。

### 3.2 性能自动降级阈值=20fps,体验先崩再降

[index.html:1401](file:///c:/Users/sunjunjie/Desktop/作业/111/sanxingdui-ar-tracker/index.html#L1401) `if (bloomEnabled && fps < 20)`。意味着要先用 <20fps 卡满 1 秒才触发关 bloom。集显笔记本上,两个 WebGL 上下文(FaceLandmarker GPU + Three.js 渲染)+ bloom + 4 个 shader + 粒子,很容易掉到 15-19fps 卡 1 秒,用户体验已经崩了才降级。降级阈值应是 25fps 触发、或用帧时间 p99(>50ms)触发,而不是等 fps 计数器走完一秒。

### 3.3 point 手势在 15Hz 检测下会丢动作

HandLandmarker 隔帧检测([index.html:1305](file:///c:/Users/sunjunjie/Desktop/作业/111/sanxingdui-ar-tracker/index.html#L1305) `handFrameToggle = !handFrameToggle`),30fps 下手势实际采样 15Hz。"指向"是快速瞬时动作,15Hz 采样 + 500ms 冷却,用户快速指一下可能根本没采到那一帧。

### 3.4 渲染错误降级后永远回不来

[index.html:1385-1393](file:///c:/Users/sunjunjie/Desktop/作业/111/sanxingdui-ar-tracker/index.html#L1385) 一旦渲染抛错,`bloomEnabled = false` 永久置 false,之后即使 WebGL 上下文恢复也不会重新启用 bloom。整个会话期间画质被锁死在降级态,无回升路径。

### 3.5 缩放系数会被手部干扰

[index.html:1099-1110](file:///c:/Users/sunjunjie/Desktop/作业/111/sanxingdui-ar-tracker/index.html#L1099) 用 `faceLandmarks` 包围盒宽度算 scale。当用户把手伸到脸前做手势时,FaceLandmarker 仍检测到脸(脸没被完全遮挡),但包围盒计算的是归一化坐标的 min/max——如果手部关键点混入(实际上 faceLandmarks 只含脸部点,不会含手),但脸部点本身在用户低头做手势时会因脸部角度变化导致包围盒跳变,scale 抖动。代码无任何对 scale 的中值滤波或异常值剔除。

### 3.6 没有任何错误恢复的用户入口

摄像头被占用、模型加载失败、WebGL 崩溃——全靠 try-catch 吞掉打 console.warn。界面上除了一个"重试"按钮(只重试启动流程),运行中崩溃无任何用户可见反馈,只有 HUD 一行小字变红。普通用户不知道发生了什么、不知道该怎么办。

## 4. 依赖与工程配置

### 4.1 `package.json` 的 `test` 是占位

```json
"test": "echo \"Error: no test specified\" && exit 1"
```

零自动化测试。对一个 1300 行逻辑、多状态机、多阈值的项目,这是工程债。手势识别算法(`isOpenPalm`/`isFist`/`isPinch` 这种纯函数)完全可以用录制的 landmarks 数据做单元测试,但没有。

### 4.2 模型从 Google Storage CDN 拉取,无本地兜底

[index.html:512](file:///c:/Users/sunjunjie/Desktop/作业/111/sanxingdui-ar-tracker/index.html#L512) `modelAssetPath: 'https://storage.googleapis.com/mediapipe-models/...'`。评委现场如果网络不稳/被墙,模型加载直接挂。wasm 已经本地化(`public/mediapipe-wasm/`),但模型权重没有本地副本。这是个单点失败风险。

### 4.3 无构建优化,单 chunk 645KB

`npx vite build` 输出 `index-XXX.js 645.71 kB │ gzip: 174.30 kB`。three.js + mediapipe + gsap 全打一个 chunk,首屏要下 174KB gzip。无 code-splitting、无懒加载。对一个 AR 应用,首屏白屏时间直接影响评委第一印象。

## 7. 终审结论

这个项目的**单点技术细节**(delegate 分配、速度预测、bloom 阈值、闪烁治理)明显是有经验的工程师写的,不是新手作业。但**整体工程姿态**是原型优先、功能半成品、文案跑在代码前面。

最致命的是 §1:手势系统识别了 4 种,只有 2 种有效,另外 2 种(张掌/握拳)识别完什么都不做;神力降临特效系统被删干净,但认主完成文案还在承诺"召唤神力降临"。**用户照着提示做动作,什么反馈都没有**——这是体验上的直接翻车,不是优化项。

次致命的是 §2:1412 行单 HTML,零模块化、零测试、全局状态散落,任何后续维护和迭代都要在 1300 行 JS 里打滚。当前还能跑,加第 5 个手势、加第 2 个面具模型就会失控。

**给分**:作为能跑的 AR 原型,70 分。作为要拿出去演示的产品,不及格——因为演示时用户一定会按提示张掌,然后什么都没有,场面会尴尬。**P0 的三条改完,这个项目就能从"翻车边缘"到"可演示"**。现在这个状态,赌的是评委不亲手操作、只看作者演示。
